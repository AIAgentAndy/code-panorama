# AI 编程实战：从零构建「代码全景图」（5 天速成版）

> 5 天时间，每天交付一个**可运行版本**，逐步叠加功能，最终完成一个 AI 驱动的代码仓库可视化分析工具。在实战中掌握 AI 编程协作和提示词工程。

---

## 课程定位

- **目标学员**：有基本编程经验（了解 HTML/CSS/JS），希望掌握 AI 辅助编程能力的开发者
- **核心收获**：AI 编程协作能力 > 提示词工程 > 全栈项目实战经验
- **最终产出**：一个可运行的「代码全景图」工具，能分析 GitHub 仓库并生成交互式函数调用链可视化
- **节奏**：每天一个可用版本，当天跑通当天验收

---

## 版本演进路线

```
Day1 V0.1 静态骨架        输入 GitHub URL → 验证仓库 → 文件列表（日志全程记录操作）
      ↓
Day2 V0.2 AI 分析引擎     入口检测 → 逐级下钻调用链（日志追踪 AI 每一步推理）
      ↓
Day3 V0.3 可视化图谱      交互式函数调用链流程图（日志可展开查看 AI 原始请求）
      ↓
Day4 V0.4 完整分析流      模块着色 + 全景文档导出（日志增加 Token 统计）
      ↓
Day5 V1.0 完整产品        历史记录 + 本地分析 + 体验打磨
```

**日志贯穿设计**：日志组件在 Day 1 就建立，随着功能叠加不断丰富日志类型——这是一种「可观测性优先」的开发思维，让你在开发过程中始终能看清系统在做什么。

```
Day1  [info] 仓库验证成功  [info] 文件树已获取 (128 个文件)  [error] URL 格式错误
Day2  [info] 入口检测: src/main.ts  [ai_request] 函数下钻分析  [ai_response] 返回 5 个子调用
Day3  [info] 图谱渲染完成 (12 节点)  可展开查看 AI 请求/响应完整 JSON
Day4  [info] 模块划分完成 (5 个模块)  [stats] 本次分析: 8 次 API 调用, 12,450 tokens
Day5  [info] 历史记录已保存  [info] 本地目录分析启动
```

## 教学理念

每天的学习遵循统一节奏：

```
明确目标 → 分析思路 → 编写提示词 → AI 生成代码 → 审查调整 → 验证运行
```

**核心原则**：不是让 AI 一次生成全部代码，而是学会**拆解问题、分步引导、迭代优化**。

## Day 1：V0.1 静态骨架 —— 从零到能跑（含日志系统）

> 今日交付：输入 GitHub URL → 验证仓库 → 展示文件树。右侧日志实时记录每一步操作。一个完整的前后端闭环。

**学习目标**
- 用 AI 从零搭建全栈项目
- 理解技术选型背后的思考
- **建立「可观测性优先」的开发习惯**：先有日志，再写功能

**实战任务**
1. 与 AI 对话描述项目愿景，让 AI 帮你初始化 Next.js + TypeScript + Tailwind CSS 项目
2. 搭建三栏布局：左侧输入区 | 中间可视化区（暂时留白） | 右侧日志区
3. **实现日志基础设施**（Day 1 最重要的基础组件）：
   - 定义日志数据结构 `LogEntry`：`{ timestamp, type, message, detail? }`
   - type 类型预留齐全：`info` / `success` / `error` / `warning` / `ai_request` / `ai_response`
   - 实现 `addLog()` 方法，供后续所有模块调用
   - 实现 AgentLog 组件：按时间倒序排列，不同 type 用不同颜色标记，自动滚动到最新
4. 实现深色/浅色主题切换
5. 实现三个后端 API Route：
   - `/api/github/validate` —— 验证仓库是否存在
   - `/api/github/tree` —— 获取完整文件树
   - `/api/github/content` —— 读取指定文件内容
6. 前端实现 GitHub URL 输入组件，点击「分析」后调用接口，左侧展示文件列表
7. 实现 LLM 设置面板（API Key、Base URL、模型名称，存入 localStorage）
8. **关键：每个操作都写入日志**，让右侧面板实时可见：
   - 点击分析 → `[info] 开始验证仓库: facebook/react`
   - 验证成功 → `[success] 仓库验证通过，默认分支: main`
   - 验证失败 → `[error] 仓库不存在或无权访问`
   - 获取文件树 → `[info] 正在获取文件树...` → `[success] 文件树获取完成，共 342 个文件`
   - 文件过滤 → `[info] 过滤后剩余 128 个代码文件`

**提示词工程 - 技巧 1：上下文铺设（Context Setting）**

> 不好的提示词：
> 「帮我创建一个 React 项目」
>
> 好的提示词：
> 「我要创建一个代码分析可视化工具。技术栈：Next.js 15 + React 19 + TypeScript + Tailwind CSS 4。项目需要前后端一体，后端用 Next.js API Routes。请帮我初始化项目，并创建一个三栏布局的首页——左栏放输入控件，中间放可视化画布（先用占位 div），右栏放实时日志列表。使用 react-resizable-panels 实现可拖拽分栏。
>
> 日志系统是核心基础设施，请优先实现：定义 LogEntry 类型（含 timestamp, type: info|success|error|warning|ai_request|ai_response, message, detail?），实现 AgentLog 组件（按时间排列、自动滚动、不同类型颜色区分），并暴露一个 addLog 方法。后续所有功能都会往这个日志里写入过程信息。」

**提示词工程 - 技巧 2：角色 + 约束条件（Role + Constraints）**

> 「你是一个 Next.js 后端开发专家。请实现 API Route `/api/github/validate`。
> 功能：接收 POST `{ url, token? }`，解析 owner/repo，调用 GitHub API 验证仓库。
> 要求：
> (1) 使用 axios
> (2) 支持可选 GitHub Token
> (3) 统一返回格式 `{ success, data?, error? }`
> (4) TypeScript 严格类型
> (5) 兼容多种 URL 格式：github.com/owner/repo、带 .git 后缀、带子路径」

**为什么这样做**
- 为什么选 Next.js？→ 需要后端 API 路由做代理，保护 API 密钥，避免 CORS
- 为什么三栏布局？→ 分析工具经典交互：输入 → 过程可视化 → 结果日志
- **为什么日志放 Day 1？→ 日志是你的「调试眼睛」。后续每加一个功能，对应的日志也一起加上，你能清楚看到系统做了什么、哪里出了问题。这是工程师最重要的习惯之一。**
- 为什么不在前端直接调 GitHub API？→ Token 安全 + CORS + 频率控制

**今日验收 V0.1**

- [ ] `npm run dev` 启动，三栏布局正常显示
- [ ] **右侧日志区正常工作，不同类型有颜色区分**
- [ ] 输入 GitHub URL 点击分析，左侧出现文件列表
- [ ] **日志面板实时显示：验证 → 获取文件树 → 过滤，每一步都能看到**
- [ ] 无效 URL 时日志显示红色错误信息
- [ ] 设置面板能配置 LLM 参数

## Day 2：V0.2 AI 分析引擎 —— 让 AI 读懂代码

> 今日交付：点击「分析」→ AI 自动识别入口 → 逐级下钻函数调用链 → 右侧日志完整追踪 AI 的每一步推理。

**学习目标**
- 封装 LLM 调用管线，让 AI 可靠返回结构化 JSON
- 实现项目入口文件的自动检测
- 掌握项目的核心算法：递归式函数调用链分析 + 多策略函数定位
- **日志进化**：从 Day 1 的操作记录，升级为 AI 推理过程的完整追踪

**实战任务**
1. 封装 LLM 调用层：
   - 后端 `/api/llm/json` 接口，转发提示词到 OpenAI 兼容 API
   - JSON 容错提取（处理 markdown 代码块、控制字符、花括号定位）
   - Token 用量统计
2. 实现文件过滤（排除 node_modules、.git、二进制文件、lock 文件）
3. 实现入口检测：规则匹配（main、index、__main__）+ 语言/框架推断
4. 调用 LLM 生成项目概要
5. **日志新增 AI 类型**：每次 LLM 调用写入两条日志
   - `[ai_request] 项目概要分析 → 发送提示词 (1,240 tokens)`
   - `[ai_response] 项目概要分析 → 返回结果 (356 tokens, 2.1s)`
6. 设计并实现核心分析 Hook（`useGithubAgent`），包含：
   - 状态机：idle → validating → fetching_tree → analyzing_structure → recursive_drilling → complete
   - 三级缓存：文件内容缓存、函数定位缓存、下钻分析缓存
7. 实现「函数下钻分析」：给 LLM 一个函数的代码片段，让它提取关键调用
8. 实现函数定位的多级策略（这是最关键的工程设计）：
   - Step 1：根据 import 推断的文件直接查找
   - Step 2：同文件回退（子函数可能在调用者同一文件）
   - Step 3：AI 基于文件列表猜测 Top 3 候选
   - Step 4：代码搜索（grep）
   - Step 5：AI 兜底读取候选文件内容后判断
9. 实现递归下钻（深度控制 + 子节点数量限制）
10. **日志丰富化**：函数定位的每一步尝试都写入日志
    - `[info] 开始定位函数: createApp`
    - `[info] step1-猜测文件: src/app.ts → 命中`
    - 或 `[info] step1-猜测文件: src/app.ts → 未命中`
    - `[info] step2-AI猜测Top3: src/core.ts, src/init.ts, lib/bootstrap.ts`
    - `[ai_request] 函数下钻分析 L1: createApp`
    - `[ai_response] 返回 5 个子调用: [initRouter, loadPlugins, ...]`
    - `[warning] 识别为系统库函数: fs.readFileSync，跳过下钻`

**提示词工程 - 技巧 3：输出格式控制（Output Format Control）**

> 需要 AI 返回结构化数据时，必须给出**完整 JSON 示例**，明确每个字段的含义和取值范围：
>
> ```
> 你是项目概要分析器。请根据 README、入口文件和清单文件，输出项目中文简介与技术栈。
>
> 仓库：{repoName}
> 语言：{language}
> 文件内容：{精心挑选的 3-6 个关键文件}
>
> 返回 JSON：
> { "summary": "中文项目概要（2-3句话）", "techStack": ["React", "TypeScript"] }
> ```
>
> **黄金法则**：不传全部信息，选择最有信息量的几个文件（README、package.json、入口文件）。

**提示词工程 - 技巧 4：结构化角色指令（Structured Instructions）**

> 这是本项目**最核心的提示词**，值得反复打磨：
>
> ```
> 你是一个"逐级下钻"的代码调用链分析器，输出必须是 JSON。
> 请分析下面文件中的目标函数（仅输出关键调用节点，忽略琐碎辅助函数）。
>
> 规则：
> 1. 只关注核心功能流程，最多输出 {N} 个子节点。
> 2. 每个子节点返回 shouldDrill：-1（叶子/不重要）、0（不确定）、1（值得继续下钻）。
> 3. 如果能推断出子函数定义文件，请填 possibleFile（项目相对路径）。
> 4. 描述要简短中文。
>
> 项目语言：{language}
> 当前文件：{filePath}
> 目标函数：{functionName}
>
> 文件头部（含 import/include）：
> {fileHead}
>
> 目标函数附近代码片段：
> {snippet}
>
> 返回 JSON：
> {
>   "functionName": "...",
>   "description": "当前函数职责简述",
>   "calls": [
>     { "name": "子函数名", "importance": "high|medium|low",
>       "shouldDrill": 1, "possibleFile": "src/xxx.ts" }
>   ]
> }
> ```
>
> **核心拆解**：
> - **角色定义**：「逐级下钻的分析器」让 AI 理解它的工作模式
> - **行为约束**：编号规则，限制输出数量
> - **上下文分离**：文件头部（import）和函数代码分开传，各有用途
> - **输出模板**：完整的 JSON 示例结构，不留歧义

**提示词工程 - 技巧 5：逃生出口（Escape Hatch）**

> 让 AI 做函数定位时，给它一个说「找不到」的选项，避免瞎编：
>
> ```
> 你是代码定位助手。请猜测该函数最可能定义在哪 3 个文件中。
> 如果是系统/标准库/第三方库函数，不要猜测项目文件，直接输出系统库标记。
>
> 返回 JSON：
> { "isSystemOrLibrary": false, "topFiles": ["a.ts", "b.ts", "c.ts"] }
> 或：
> { "isSystemOrLibrary": true, "reason": "这是 Node.js 内置 fs 模块函数" }
> ```

**为什么这样做**
- 逐级下钻 vs 一次分析全部？→ Token 限制 + 聚焦分析质量更高
- shouldDrill 三值设计？→ -1 确定叶子、0 不确定、1 明确重要，控制下钻广度
- 多策略定位的设计哲学？→ 从「成本低、确定性高」到「成本高、确定性低」逐级 fallback
- 缓存？→ 同一工具函数被多处调用，不重复分析

**今日验收 V0.2**

- [ ] 点击分析后，AI 自动检测入口文件
- [ ] 递归下钻函数调用链（可配置深度）
- [ ] **日志完整追踪 AI 推理**：每次 LLM 调用显示 `[ai_request]` 和 `[ai_response]`
- [ ] **日志追踪函数定位**：能看到每个函数定位的逐步尝试（step1 → step2 → ...）
- [ ] **系统库函数有 warning 日志**，不会继续无意义下钻
- [ ] 分析完成后控制台可看到完整的调用链数据结构

## Day 3：V0.3 可视化图谱 —— 数据变成流程图

> 今日交付：分析结果不再只是 JSON，而是一张**可交互的函数调用链流程图**。日志面板升级为可展开查看 AI 原始请求/响应。

**学习目标**
- 用 AI 辅助开发交互式图谱组件
- 理解图布局算法的基本原理
- **日志进化**：AI 请求/响应日志可展开查看完整 JSON

**实战任务**
1. 安装 @xyflow/react，实现 GraphViewer 组件
2. 将调用链数据（nodes + edges）转换为 ReactFlow 格式
3. 实现树形布局算法：
   - 计算每个节点的深度（BFS）
   - 用 DFS 排列 Y 坐标
   - 从左到右层级展开
4. 实现自定义节点组件 CustomNode
5. 实现节点点击 → 右侧显示函数详情（文件路径、代码片段、描述）
6. **日志组件升级**（在 Day 1 基础上增强）：
   - `[ai_request]` 和 `[ai_response]` 条目**可展开/收起**，查看完整 JSON
   - 点击日志条目高亮关联的图谱节点
   - 日志区和图谱区联动：点击图谱节点，日志自动滚动到该节点的分析记录
7. 底部状态栏显示 AI 用量统计（总 Token 数、API 调用次数）
8. 实现缩放控制按钮（放大、缩小、适配画布）

**提示词工程 - 技巧 6：组件化提示词（Component-Oriented Prompts）**

> 复杂 UI 拆成小组件，逐个让 AI 实现，用编号确保不遗漏：
>
> 「请用 @xyflow/react 实现自定义节点组件 CustomNode。要求：
> (1) 显示函数名（label）作为标题
> (2) 右上角显示类型标签（function/class/module），用不同颜色
> (3) 下方小字显示文件路径（file:line）
> (4) 左侧一条彩色竖线，颜色由所属模块 prop 决定
> (5) importance="high" 时标题加粗
> (6) selected 状态加高亮边框
> (7) Tailwind CSS，支持 dark 模式
> (8) 底部一个展开按钮，hasChildren=false 时显示，点击触发 onExpand」

**提示词工程 - 技巧 7：迭代优化（Iterative Refinement）**

> AI 生成结果不够好时，不要推倒重来，**追加约束**：
>
> 第一轮：「分析这个函数的调用链」
> → 问题：返回了太多琐碎调用
>
> 第二轮追加：「**只关注核心流程，忽略日志、异常处理、工具函数，最多 10 个**」
> → 问题：框架启动函数被当作终点
>
> 第三轮追加：「**框架启动调用（如 SpringApplication.run）标记为桥接节点 shouldDrill=1**」
>
> 每次根据 AI 实际输出调整约束，这就是提示词的迭代优化。

**为什么这样做**
- 为什么用 ReactFlow 而不是 D3？→ ReactFlow 开箱自带拖拽、缩放、连线，AI 也更熟悉
- 为什么从左到右布局？→ 调用链有方向，从入口到叶子符合阅读习惯

**今日验收 V0.3**

- [ ] 分析完成后中间区域显示交互式流程图
- [ ] 节点显示函数名、文件路径、类型标签
- [ ] 支持缩放、拖拽
- [ ] 点击节点右侧显示详情
- [ ] **日志中 AI 请求/响应条目可展开查看完整 JSON**
- [ ] **底部状态栏显示累计 Token 数和 API 调用次数**

8. 实现节点收起/展开（隐藏/显示子树）

**提示词工程 - 技巧 8：全局视角提示词（Global Context Prompts）**

> 模块划分需要 AI 看到**全局**后做决策，上下文必须充足：
>
> ```
> 你是代码架构模块划分专家。请根据项目概述、文件列表、完整函数调用链，
> 对全景图节点进行模块划分。
>
> 要求：
> 1. 按"职责/领域"划分（不是按目录）。
> 2. 模块数量 3~10 个。
> 3. 所有 nodeId 必须分配到一个模块。
>
> 项目元数据：{metadata}
> 项目概要：{summary}
> 完整函数调用链：{callChain}
> 当前图节点：{nodes 列表}
>
> 返回 JSON：
> {
>   "modules": [{ "id": "module_core", "name": "核心流程" }],
>   "nodeModuleMap": { "node_id_1": "module_core" }
> }
> ```
>
> **关键**：用数量约束（3~10）避免 AI 过度细分或过度合并。

**提示词工程 - 技巧 9：AI 作为设计伙伴（Design Partner）**

> 先让 AI 帮你设计数据格式，再写代码：
>
> 「我的代码分析工具每次产出：项目元信息、概要、文件列表、函数调用链图。
> 我想序列化为 Markdown 文档，同时能被程序反序列化。
> 请设计这个 Markdown 结构。要求：(1) 人类可读 (2) 程序可解析 (3) 调用链用 JSON 代码块嵌入。」
>
> **核心**：先设计后编码。AI 是很好的产品设计讨论伙伴。

**提示词工程 - 技巧 10：复用思维（Reuse-Oriented Prompts）**

> 「我已经有了完整的 GitHub 仓库分析流程（validate → tree → content → search 四个接口）。
> 现在要支持分析本地目录。请帮我设计 Local 版本的四个 API，接口签名尽量与 GitHub 版本一致，
> 让前端分析引擎只需要切换一个 sourceType 参数就能复用全部逻辑。」
>
> **核心**：告诉 AI 你的已有架构，让它基于现有设计做扩展，而不是重新发明。

**方法论总结：AI 编程五步法**

经过 5 天实战，回顾你学到的 AI 编程方法论：

**第一步：明确问题（What）**
在写任何提示词之前，先想清楚：要做什么？输入是什么？输出是什么？

**第二步：拆解步骤（How）**
不要让 AI 一口吃成胖子。把大问题拆成小步骤，每步独立验证。

**第三步：精心写提示词（Prompt）**
核心技巧速查表：

| # | 技巧 | 一句话总结 | 使用场景 |
|---|------|-----------|---------|
| 1 | 上下文铺设 | 先说背景再提需求 | 项目初始化、新功能开发 |
| 2 | 角色 + 约束 | 定义 AI 身份，列出明确要求 | API 开发、工具函数 |
| 3 | 输出格式控制 | 给完整 JSON 示例模板 | 所有需要结构化返回的场景 |
| 4 | 结构化角色指令 | 编号规则 + 分区上下文 | 复杂分析任务 |
| 5 | 逃生出口 | 给 AI 说「不知道」的选项 | 不确定性高的决策场景 |
| 6 | 组件化提示词 | 拆成小组件逐个实现 | UI 开发 |
| 7 | 迭代优化 | 根据输出追加约束 | 效果不满意时 |
| 8 | 全局视角 | 传足上下文做全局决策 | 分类、架构设计 |
| 9 | AI 做设计伙伴 | 先讨论方案再编码 | 数据格式、架构设计 |
| 10 | 复用思维 | 告诉 AI 已有架构做扩展 | 功能扩展、接口适配 |

**第四步：审查代码（Review）**
AI 生成的代码不是终点。每次都要审查：安全性、边界情况、命名规范、是否过度设计。

**第五步：迭代优化（Iterate）**
第一版永远不完美。根据实际运行效果，调整提示词、修复 Bug、优化体验。
